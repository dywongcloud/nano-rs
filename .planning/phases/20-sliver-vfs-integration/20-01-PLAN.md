## Bug Analysis

The current implementation in `src/worker/pool.rs` (lines 60-61, 109-110):
```rust
let code = fs::read_to_string(&handler_ctx.entrypoint)
    .map_err(|e| anyhow!("Failed to read entrypoint: {}", e))?;
```

This reads from the OS filesystem path passed in `handler_ctx.entrypoint`. When running from a sliver:
1. The sliver contains JS files in `vfs/` (e.g., `vfs/index.js`)
2. The sliver unpacker extracts these to `vfs_entries: Vec<(VfsPath, VfsFile)>`
3. The VFS is restored to an `IsolateVfs` with memory backend
4. But the `entrypoint` is still the original OS path (e.g., `index.js` from CWD)
5. When the sliver is moved to `/tmp` without source files, `fs::read_to_string` fails

## Solution Approach: Option A (Extract to Temp)

Extract the sliver's VFS to a temp directory on load, then execute from there.

**Advantages:**
- Fast implementation (no custom module loader needed)
- Compatible with existing code structure
- File system semantics preserved (imports work naturally)

**Implementation:**
1. On sliver load: Extract `vfs/` contents to `$TMPDIR/nano-sliver-{hostname}-{random}/`
2. Set `entrypoint` to extracted path
3. Use standard `fs::read_to_string` (works because it's real filesystem)
4. Cleanup temp directory on shutdown

## VFS Structure

From `src/sliver/format.rs`:
- `VFS_PREFIX = "vfs/"` - All VFS files stored under this prefix in tar
- `vfs_entries: Vec<(VfsPath, VfsFile)>` in `UnpackedSliver`
- Paths like `vfs/index.js`, `vfs/utils/helpers.js`

## Temp Directory Design

```
$TMPDIR/nano-sliver-{hostname}-{random}/
├── index.js          # entrypoint
├── utils/
│   └── helpers.js    # imported modules
└── assets/
    └── logo.png      # static assets
```

## Security Considerations

- Use `tempfile` crate for secure temp directory creation
- Restrict permissions to owner only (0o700)
- Clean up on shutdown (even on panic)
- Unique temp dir per sliver instance (prevents collisions)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sliver → temp | VFS content extracted to host temp filesystem |
| temp → worker | Workers read JS from temp (same trust level) |
| multi-tenant | Each sliver gets isolated temp directory |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | Tampering | VFS extraction | mitigate | Use `tempfile::Builder` with secure permissions (0o700) |
| T-20-02 | Information Disclosure | Temp directory | mitigate | Owner-only permissions, cleanup on shutdown |
| T-20-03 | Denial of Service | Temp disk space | mitigate | Limit total temp size (VFS size known), cleanup guaranteed |
| T-20-04 | Elevation | Path traversal in VFS | mitigate | VfsPath validation prevents traversal (../) in entry names |
| T-20-05 | Repudiation | Cleanup audit | accept | Log cleanup operations; temp content is ephemeral anyway |
</threat_model>
