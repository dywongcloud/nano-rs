## Current Architecture (from Phase 999.2)

The WorkerPool trait architecture is now consolidated:
- **EntrypointWorkerPool** (queue.rs): Entrypoint dispatch with async VFS creation
- **WorkQueue** (queue.rs): Multi-hostname pool manager with `get_or_create_pool()`
- **AppRegistry** (registry.rs): Stores per-app configurations including `vfs_backend` and `vfs_disk`

## The Gap

WorkQueue currently has a global `vfs_disk_config: Option<VfsDiskConfig>` field. When `get_or_create_pool()` creates a new pool, it only checks this global config - NOT the per-app configuration from AppRegistry.

**E2E test config that needs to work:**
```json
{
  "apps": [{
    "hostname": "wasm-normal.local",
    "entrypoint": "/abs/path/wasm_normal.js",
    "vfs_backend": "disk",
    "vfs_disk": { "base_path": "/tmp/test_dir" }
  }]
}
```

**File structure expected by DiskBackend:**
```
/tmp/test_dir/
├── wasm_normal.js          # Entrypoint (read directly by runtime)
└── wasm_normal_local/       # Sanitized hostname dir (VFS-accessible)
    └── add.wasm            # WASM file (read via Nano.fs)
```

**Hostname sanitization:** `wasm-normal.local` → `wasm_normal_local` (dots/hyphens → underscores)

## Solution Approach

Option A (Recommended): Pass AppRegistry to WorkQueue
- Add `app_registry: Option<Arc<AppRegistry>>` to WorkQueue
- Modify `get_or_create_pool()` to look up app config from registry
- Create DiskBackend with per-app config when `vfs_backend == Disk`
- Fallback to MemoryBackend if no registry or no disk config

Option B: Pass VFS config through AppState
- Modify AppState constructors to pass per-app VFS configs
- More complex - requires restructuring how pools are created

**Selected: Option A** - WorkQueue already has registry access via AppState, just needs to receive and use it.

<task type="auto">
  <name>Task 2: Modify get_or_create_pool to use per-app VFS config</name>
  <files>src/worker/queue.rs</files>
  <action>
Modify `get_or_create_pool()` method (lines 434-470) to use per-app VFS configuration from AppRegistry instead of global `vfs_disk_config`:

**Current logic (lines 440-460):**
```rust
let pool = if let Some(ref disk_config) = self.vfs_disk_config {
    // Create disk backend from global config
} else {
    EntrypointWorkerPool::new(hostname, self.workers_per_pool)
};
```

**New logic:**
```rust
// Check for per-app VFS configuration from AppRegistry
let pool = if let Some(ref registry) = self.app_registry {
    if let Some(app_config) = registry.get(hostname) {
        match app_config.vfs_backend {
            VfsBackendType::Disk => {
                if let Some(ref disk_config) = app_config.vfs_disk {
                    // Create disk backend with per-app config
                    match BackendFactory::new()
                        .create_backend(VfsBackendType::Disk, Some(disk_config), None)
                        .await
                    {
                        Ok(backend) => {
                            tracing::info!("Created disk backend for hostname: {} with base_path: {}", 
                                hostname, disk_config.base_path);
                            EntrypointWorkerPool::with_backend(hostname, self.workers_per_pool, backend)
                        }
                        Err(e) => {
                            tracing::warn!("Failed to create disk backend for {}, falling back to memory: {}", 
                                hostname, e);
                            EntrypointWorkerPool::new(hostname, self.workers_per_pool)
                        }
                    }
                } else {
                    tracing::warn!("App {} has vfs_backend=disk but no vfs_disk config, using memory", hostname);
                    EntrypointWorkerPool::new(hostname, self.workers_per_pool)
                }
            }
            VfsBackendType::Memory | _ => {
                tracing::debug!("Using memory backend for hostname: {}", hostname);
                EntrypointWorkerPool::new(hostname, self.workers_per_pool)
            }
        }
    } else {
        // App not found in registry, use memory backend
        EntrypointWorkerPool::new(hostname, self.workers_per_pool)
    }
} else {
    // No registry available, use memory backend (backward compatible)
    EntrypointWorkerPool::new(hostname, self.workers_per_pool)
};
```

**Important:** 
- Import VfsBackendType at top of file (check if already imported)
- Ensure the logic gracefully falls back to MemoryBackend for any error case
- Add appropriate tracing logs for debugging

| Config → AppRegistry | Untrusted user config deserialized into AppConfig |
| AppRegistry → WorkQueue | Internal boundary, both trusted code |

| DiskBackend → OS Filesystem | Actual file I/O with path sanitization |

| T-999.3-02 | Elevation | DiskBackend path | mitigate | DiskBackend::sanitize_namespace() already prevents traversal |
| T-999.3-03 | DoS | Per-app backend creation | accept | Existing limits: bounded channels, timeout enforcement |
| T-999.3-04 | Info Disclosure | Registry access | accept | AppRegistry is internal, no sensitive data exposed |

**Note:** The primary security mechanisms (path sanitization, config validation) already exist in the codebase. This phase just wires them together differently.
