<objective>
Implement production-grade CPU time limits with timer-based termination using Linux timer_create syscall and V8 TerminateExecution.

**Purpose:** Production multi-tenancy requires strict CPU limits per request to prevent runaway scripts from starving other tenants. Cloudflare uses 50ms CPU time limits; we implement the same pattern with microsecond-precision tracking.

**Output:** CPU tracking infrastructure, timer-based termination, and per-app configurable limits.
</objective>

## Technical Context

**Current Implementation:**
- `src/worker/limits.rs` - Memory limits with HeapStatistics
- `src/worker/pool.rs` - WorkerPool with thread-local isolates
- `src/config/app.rs` - AppLimits with memory_mb, timeout_secs, workers
- Timeout exists but is wall-clock based (tokio::time::timeout)

**Required Approach (Cloudflare-style):**
1. CPU time tracking via `timer_create(CLOCK_THREAD_CPUTIME_ID)` or `thread_time` crate
2. Per-request CPU limit (default 50ms like Cloudflare Workers)
3. Timer signal handler calls V8's `Isolate::TerminateExecution()`
4. Distinguish CPU timeout from wall-clock timeout

**V8 Integration:**
```rust
// Terminate execution from signal handler
isolate.terminate_execution();
// Later check: isolate.is_execution_terminating()
```

**Rust Crates:**
- `thread-time` - Cross-platform thread CPU time (Linux uses clock_gettime)
- `signal-hook` - Safe signal handling for SIGALRM
</context>

  <name>Task 2: Implement timer-based execution termination</name>
  <files>src/worker/timeout.rs, src/worker/pool.rs, Cargo.toml</files>
  <action>
Create `src/worker/timeout.rs` with Linux timer_create-based termination:

1. **TimeoutConfig struct:**
   ```rust
   pub struct TimeoutConfig {
       pub cpu_time_limit_ms: u32,  // 50ms default
       pub wall_clock_limit_ms: u32, // 30s default (backup)
       pub termination_grace_us: u32, // 100us grace period
   }
   ```

2. **ExecutionTimer struct:**
   - Use `timer_create(CLOCK_THREAD_CPUTIME_ID, ...)` on Linux
   - Fallback to tokio::time::timeout on other platforms
   - SIGALRM handler that calls isolate.terminate_execution()
   - Safe signal handling with signal-hook

3. **Implementation approach:**
   ```rust
   pub struct ExecutionTimer {
       config: TimeoutConfig,
       cpu_tracker: CpuTracker,
       timer_id: Option<timer_t>, // POSIX timer
   }
   
   impl ExecutionTimer {
       pub async fn run_with_timeout<F, T>(
           &self,
           isolate: &mut v8::Isolate,
           f: F
       ) -> Result<T, TimeoutError>
       where F: Future<Output = T> { ... }
   }
   ```

4. **Signal handler safety:**
   - Use signal-hook's async-safe channels
   - Handler sets atomic flag, main thread calls terminate_execution()
   - Never call V8 from signal handler directly

5. **Integration into pool.rs:**
   - Wrap handler execution in ExecutionTimer
   - Check CPU time after each JS call
   - Terminate if limit exceeded

**Platform support:**
- Linux: timer_create(CLOCK_THREAD_CPUTIME_ID)
- macOS: thread_time crate (mach thread_info)
- Windows: QueryThreadCycleTime (future)

**Add to Cargo.toml:**
```toml
[target.'cfg(target_os = "linux")'.dependencies]
libc = "0.2"
nix = { version = "0.29", features = ["signal"] }

[dependencies]
signal-hook = "0.3"
```

**Key safety rule:** The signal handler MUST NOT call V8 directly. Set atomic flag, check in main execution loop.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| JS code → CPU tracker | Untrusted user code CPU consumption |
| Signal handler → V8 | Signal handler must not call V8 directly |
| Timer thread → Isolate | Timer must not violate thread-safety |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-01 | Denial of Service | CPU tracker | mitigate | Per-request limits prevent infinite loops |
| T-27-02 | Elevation of Privilege | Signal handler | mitigate | Atomic flag only, no V8 calls from handler |
| T-27-03 | Information Disclosure | CPU timing | accept | Timing info not exposed to JS |
| T-27-04 | Tampering | Timeout config | mitigate | Config validated at load time |
| T-27-05 | Denial of Service | Timer resource exhaustion | mitigate | One timer per isolate, cleaned up on drop |
