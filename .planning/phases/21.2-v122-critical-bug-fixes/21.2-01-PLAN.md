# Plan 21.2-01: Fix VFS Path Validation for Special Characters

## Goal
Fix the VFS path validation to allow legitimate file patterns containing `[...]` characters (used by Astro, Next.js, and other frameworks for catch-all and optional routes) while still preventing actual path traversal attacks.

## Background
The VFS security validator currently rejects any path containing `..` anywhere in the string, which incorrectly blocks valid filenames like:
- `[...slug].astro` (Astro catch-all routes)
- `[[...optional]].ts` (Astro optional catch-all)
- `[...path].js` (Next.js dynamic routes)

The error message shows:
```
EINVAL: Path contains '..' which is not allowed: /examples/astroblog/src/pages/blog/[...slug].astro
```

## Requirements

### REQ-21.2-01-01: Allow Square Brackets in Filenames
VFS path validation must accept filenames containing `[`, `]`, and `...` characters when they are part of the filename (not path traversal).

### REQ-21.2-01-02: Prevent Path Traversal
The fix must continue to block actual path traversal attempts like:
- `../../../etc/passwd`
- `foo/../bar` (directory traversal)
- `..` as a path segment

### REQ-21.2-01-03: Test Coverage
All existing VFS tests must continue to pass, and new tests must be added for special character filenames.

## Implementation

### 1. Locate and Fix Path Validation
**File:** `src/vfs/types.rs` (lines 48-54)

**Current code:**
```rust
if normalized.contains("..") {
    return Err(VfsError::InvalidPath {
        path: path.to_string(),
        reason: "Path contains '..' which is not allowed".to_string(),
    });
}
```

**Fixed code:**
```rust
// Check for traversal attempts after normalization
// Only reject ".." as a full path segment, not as part of filenames
if normalized.split('/').any(|segment| segment == "..") {
    return Err(VfsError::InvalidPath {
        path: path.to_string(),
        reason: "Path contains '..' segment which is not allowed".to_string(),
    });
}
```
