       Inside handle_ws_upgrade:
       a. Extract the WebSocketUpgrade from the request. Since we are NOT in an axum handler signature, use WebSocketUpgrade::from_request(request, &amp;()) — this consumes the request. If extraction fails (malformed upgrade), return 400 Bad Request response.

       b. Resolve the tenant/entrypoint for this host using the same lookup logic as the HTTP path (state.router, state.work_queue). Extract the entrypoint path, cpu_time_limit_ms, memory_limit_mb from the resolved route config.

          - let (inbound_tx, inbound_rx) = std::sync::mpsc::sync_channel::&lt;tungstenite::Message&gt;(128);
          - let (outbound_tx, outbound_rx) = std::sync::mpsc::sync_channel::&lt;tungstenite::Message&gt;(128);

       e. Build WsChannels { inbound_rx, outbound_tx } and construct a HandlerTask with ws: Some(ws_channels). The NanoRequest for the HandlerTask should capture method, URL, and headers from the original request (before WS extraction consumes it — so extract these BEFORE calling WebSocketUpgrade::from_request, or use request.uri() and request.headers() cloned beforehand).

       f. Dispatch the HandlerTask to the work queue / tenant pool. If dispatch fails due to connection limit (dispatch_ws returns error), return 503 with "WebSocket connection limit reached" body and "Retry-After: 1" header per D-07.

       g. Call ws_upgrade.on_upgrade(move |socket| ws_relay_task(socket, inbound_tx, outbound_rx)). The on_upgrade callback returns the 101 response to the client. Convert the axum response to Response&lt;Body&gt; for the return type.

    4. Define async fn ws_relay_task(mut socket: WebSocket, inbound_tx: std::sync::mpsc::SyncSender&lt;tungstenite::Message&gt;, outbound_rx: std::sync::mpsc::Receiver&lt;tungstenite::Message&gt;). This is the tokio task that bridges async WS stream to sync channels per D-05b:

       a. The relay runs two concurrent loops using tokio::select!:
          - Inbound: socket.recv().await — convert AxumWsMessage to TungsteniteMessage, enforce 32 MiB limit (D-09/D-12b). If message payload exceeds 32 * 1024 * 1024 bytes, send close frame 1009 via socket and break. Otherwise send to inbound_tx. If inbound_tx.send() fails (worker dropped receiver), break.
          - Outbound: spawn a tokio::task::spawn_blocking that calls outbound_rx.recv() in a loop (since outbound_rx is std::sync::mpsc::Receiver, it blocks). Each received TungsteniteMessage is sent back via a tokio::sync::mpsc channel to the select! loop which forwards to socket.send().

       b. Alternative simpler approach for outbound: use tokio::task::spawn_blocking to run a loop reading from outbound_rx and writing to a tokio::sync::mpsc::Sender. The select! loop reads from that tokio receiver and socket simultaneously.

       c. On any error or channel close: drop inbound_tx (this signals the worker thread that the connection is gone — it will see RecvTimeoutError::Disconnected per D-17b). Close the WebSocket with appropriate frame.

       d. Ping/Pong: axum WS handles pong replies automatically per D-15b. Never forward Ping/Pong to inbound_tx.

    5. Handle the AxumWsMessage to TungsteniteMessage conversion. axum::extract::ws::Message has variants Text, Binary, Ping, Pong, Close. Map:
       - AxumWsMessage::Text(s) -> TungsteniteMessage::Text(s)
       - AxumWsMessage::Binary(b) -> TungsteniteMessage::Binary(b)
       - AxumWsMessage::Close(frame) -> TungsteniteMessage::Close(convert close frame)
       - AxumWsMessage::Ping/Pong -> skip (D-15b)

       And the reverse for outbound (TungsteniteMessage -> AxumWsMessage).
