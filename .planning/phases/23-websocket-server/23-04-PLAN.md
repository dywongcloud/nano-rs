<objective>
Extend the worker thread's run_worker function with a 'ws_messages inner loop. When a HandlerTask arrives with ws: Some(WsChannels), the worker enters the WS message loop instead of the HTTP handler path. The loop dispatches inbound frames to JS event handlers (stored as Global Function in thread-locals), enforces OOM/CPU limits per message, and handles connection close with full context reset.

Purpose: This is the core WS message processing loop per D-01 (pin-a-worker), D-03 (drain-then-recycle), D-09b (per-message CpuTimeoutGuard), D-10 (sequential per-connection), and D-13 (graceful OOM).
Output: Worker threads can process WebSocket connections end-to-end: receive frames, call JS handlers, send responses, handle close/error, recycle.
</objective>

          - Skip per D-15b. continue 'ws_messages.

          - D-17b: relay task dropped inbound_tx (stream error or client disconnect)
