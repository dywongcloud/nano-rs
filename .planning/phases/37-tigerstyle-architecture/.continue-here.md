# BLOCKING CONSTRAINTS — Read Before Anything Else

> These are not suggestions. Each constraint below was discovered through failure.
> Acknowledge each one explicitly before proceeding.

- [ ] CONSTRAINT: VFS-vs-Disk Mismatch — The runtime (`src/data_plane.rs:96`, `src/runtime/handler.rs:36`) reads entrypoints from the OS filesystem via `std::fs::read_to_string()`, NOT from VFS. Tests that write JS to VFS (`IsolateVfs` with `MemoryBackend`) and then dispatch with a VFS path like `/app.js` will fail with "No such file or directory". ALL integration tests writing JS code must use `std::env::temp_dir()` + `std::fs::write()` instead.
- [ ] CONSTRAINT: Context Reset Between Requests — The runtime resets the V8 context between requests by design (5ms, not full isolate recreation). In-memory `Map`/`const` state does NOT persist across dispatches. Multi-request tests that depend on state persistence will fail. Rewrite them as single-request tests that perform all assertions internally.
- [ ] CONSTRAINT: Config Limit Enforcement — TigerStyle limits are enforced: `timeout_secs` must be 1-30, `memory_mb` must be 1-256. Test configs and fixture JSON files must respect these bounds or `load_config()` will fail validation.
- [ ] CONSTRAINT: Adversarial Tests Hang — Tests in `tests/adversarial_*.rs` and `tests/cpu_timeout_*.rs` spawn actual nano-rs server processes and may hang or require very long timeouts (>5 minutes). These are pre-existing E2E tests, not caused by Phase 37 changes. Run them individually or skip in batch runs.

**Do not proceed until all boxes are checked.**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| VFS-only test setup | Writing test JS to `MemoryBackend` VFS and dispatching with VFS path. Runtime reads from disk, not VFS. | blocking | Always write test JS to `std::env::temp_dir()` using `std::fs::write()` |
| Multi-request state dependency | CRUD tests making sequential requests assuming state persists. Context resets between requests. | blocking | Consolidate into single request or use external storage |
| Static test assertions | Hard-coded expected strings in tests can match placeholder responses. | advisory | Use dynamic UUID tokens for response validation |
| Hanging E2E tests | adversarial_* and cpu_timeout_* tests spawn real server processes and may deadlock. | advisory | Run individually with `--test <name>` and long timeouts |

_Severity values:_ `blocking` — resuming agent must pass understanding check before proceeding. `advisory` — important context, does not gate resumption.
