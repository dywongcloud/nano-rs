### 2. Security Test Infrastructure
- **security_utils.rs**: Shared test harness with `NanoProcess` for E2E tests, V8 initialization helpers, VFS test context
- **Integration pattern**: Tests spawn real NANO processes to verify actual timeout/termination behavior

### 3. CVE Scanning Tools
- **tools/cve_scanner.rs**: CLI tool wrapping cargo-audit
  - Supports JSON/human output formats
  - Severity filtering (critical, high, medium, low)
  - Offline mode for air-gapped environments
  - Exit codes: 0=clean, 1=CVEs found, 2=error

### 4. Makefile Security Targets
```bash
make test-security          # Run adversarial tests
make test-cve-check         # Basic CVE scan
make test-cve-check-strict # Deny warnings
make security-gate          # Run both tests + CVE check
make security-scan          # Full security scan
make security-update-db     # Update CVE database
```

### 5. Security Documentation
- **docs/SECURITY_GATEWAY.md**: 500+ line comprehensive security document
  - Security guarantees for all 8 attack vectors
  - Trust boundaries diagram (ASCII art)
  - CVE response process with SLA
  - Production security checklist
  - Known limitations and mitigations
  - Security issue reporting procedures

### 6. CI Security Gate
- **.github/workflows/security.yml**: 4-job workflow
  - `security-tests`: Runs 69 adversarial tests
  - `cve-scan`: Daily CVE monitoring
  - `cargo-deny`: License/security policy enforcement
  - `security-lint`: Clippy with security-focused lints
  - Blocks PRs/releases on any security failure
  - Runs on: push to main, PRs, daily cron

### 7. Configuration Files
- **deny.toml**: cargo-deny configuration
  - License whitelist (MIT, Apache-2.0, BSD-3-Clause)
  - Security advisory deny-list
  - Duplicate crate detection
- **.cargo/audit.toml**: cargo-audit CI configuration
  - Database caching settings
  - Severity thresholds
  - Exception documentation template

## Key Design Decisions

1. **Test isolation**: Each security test spawns fresh NANO process to verify real-world termination/isolation behavior
2. **Modular structure**: Separate files per attack vector for maintainability
3. **CI integration**: Security gate blocks releases, not just warns
4. **Documentation-first**: SECURITY_GATEWAY.md provides single source of truth for security posture

## Recommendations for Future Enhancements
