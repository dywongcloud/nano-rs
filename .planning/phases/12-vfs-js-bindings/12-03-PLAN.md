# Phase 12 Plan 03: Error Handling & JS Exceptions

## Objective
Implement proper error code compatibility between VFS errors and JavaScript exceptions. Ensure Node.js-style error codes (ENOENT, EACCES, etc.) are properly propagated to JavaScript with correct messages and stack traces.

## Context
- @../../../src/vfs/types.rs — VfsError enum with error codes
- @../../../src/runtime/vfs_bindings.rs — Nano.fs JS bindings
- @../../../src/runtime/fs_polyfill.rs — Node.js fs polyfill

## Current State
VFS has VfsError types with Node.js-compatible codes:
- NotFound → ENOENT
- PermissionDenied → EACCES
- AlreadyExists → EEXIST
- InvalidPath → EINVAL
- QuotaExceeded → EQUOTA
- IoError → EIO

What's missing:
- JS Error object creation with proper code property
- Error message formatting matching Node.js
- Proper exception throwing from Rust callbacks
- Stack trace preservation in async contexts

## Success Criteria
1. VFS errors thrown as JS Error objects
2. Error.code property matches Node.js (ENOENT, EACCES, etc.)
3. Error.message matches Node.js format
4. Async methods reject Promise with Error object
5. Sync methods throw Error object
6. EQUOTA error code supported for NANO-specific quota errors

## Tasks

### Task 1: Create JS error conversion utility
**Type:** auto
**File:** src/runtime/vfs_error.rs (new)

Create utility to convert VfsError to V8 JS Error:
1. `fn vfs_error_to_js_error(scope, VfsError) -> v8::Local<v8::Value>`
2. Creates Error object with:
   - message: formatted error string
   - code: error code string (ENOENT, EACCES, etc.)
   - path: file path if available
3. Uses V8 API to create Error object
4. Add custom 'code' property to Error instance
5. Add custom 'path' property to Error instance if available

### Task 2: Update Nano.fs bindings with error handling
**Type:** auto
**File:** src/runtime/vfs_bindings.rs

1. Update each method to catch VfsError and convert to JS Error
2. For async methods:
   - On success: resolve Promise with result
   - On error: reject Promise with converted Error
3. For sync operations (if any):
   - On error: throw Error using V8 exception mechanism
4. Ensure error messages are user-friendly

### Task 3: Update fs polyfill with error handling
**Type:** auto
**File:** src/runtime/fs_polyfill.rs

1. Ensure sync methods throw JS Error on failure
2. Ensure async callbacks receive Error as first argument on failure
3. Match Node.js error message format exactly
4. Verify error codes match Node.js expectations

### Task 4: Add error code tests
**Type:** auto

Create tests for each error code:
