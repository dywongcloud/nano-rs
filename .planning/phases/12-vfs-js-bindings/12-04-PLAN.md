# Phase 12 Plan 04: Security Tests & Path Validation

## Objective
Implement comprehensive security tests for the VFS JavaScript bindings to ensure path traversal attacks are blocked, namespace isolation works correctly, and resource limits are enforced from JavaScript.

## Context
- @../../../src/vfs/security.rs — PathValidator and ResourceLimiter
- @../../../src/runtime/vfs_bindings.rs — Nano.fs API
- @../../../src/runtime/fs_polyfill.rs — Node.js fs polyfill

## Current State
Security infrastructure exists:
- PathValidator blocks traversal attempts with ".."
- ResourceLimiter enforces file size and count quotas
- Namespace isolation in IsolateVfs

What's missing:
- Comprehensive tests from JavaScript perspective
- Tests that verify traversal is blocked at JS API layer
- Tests for resource limit enforcement from JS
- Fuzzing-style tests for path edge cases

## Success Criteria
1. Path traversal blocked from both Nano.fs and Node.js fs
2. Namespace isolation verified via JS tests
3. Resource limits enforced from JS
4. Unicode and special characters handled safely
5. Null bytes rejected
6. All security tests pass

## Tasks

### Task 1: Create security test module
**Type:** auto
**File:** tests/vfs_security_tests.rs (new)

Create comprehensive security tests:
1. Path traversal tests:
   - Test `../etc/passwd` is blocked
   - Test `foo/../../etc/passwd` is blocked
   - Test `foo/../bar` is blocked
   - Test multiple slashes (`//etc/passwd`) normalized safely
   - Test null byte injection (`file\0.txt`)
2. Namespace isolation tests:
   - Test app A cannot read app B's files via any path
   - Test cross-namespace file access is blocked
3. Resource limit tests:
   - Test max file size limit from JS
   - Test max total storage limit from JS
   - Test max file count limit from JS
4. Unicode tests:
   - Test unicode paths work correctly
   - Test emoji in filenames
   - Test RTL characters

### Task 2: Create path edge case tests
**Type:** auto
**File:** tests/vfs_security_tests.rs

Test edge cases:
1. Empty paths
2. Root path "/"
3. Very long paths (near MAX_PATH_LENGTH)
4. Paths with only slashes
5. Paths with special characters (spaces, tabs, newlines)
6. Paths that look like traversal but aren't (e.g., "..file.txt")
7. Case sensitivity tests (if applicable)

### Task 3: Add integration security tests
**Type:** auto
**File:** tests/security_integration_tests.rs (new)

Create integration tests with actual HTTP requests:
1. Test that malicious paths in request handlers are blocked
2. Test that file operations from user scripts respect boundaries
3. Test with actual app configuration and WorkerPool

### Task 4: Verify error handling for security violations
**Type:** auto
**File:** tests/vfs_security_tests.rs

Ensure security violations produce appropriate errors:
1. Traversal attempts return EINVAL
2. Permission violations return EACCES
3. Quota violations return EQUOTA
4. Error messages don't leak internal paths

### Task 5: Create benchmark/security stress tests
**Type:** auto
**File:** benches/vfs_security_bench.rs (new)

Add performance/security stress tests:
1. Test rapid file operations don't bypass security
2. Test concurrent access doesn't create race conditions
3. Test memory usage under high load
