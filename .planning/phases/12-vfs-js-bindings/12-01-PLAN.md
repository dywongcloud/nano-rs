# Phase 12 Plan 01: Nano.fs.* JavaScript API

## Objective
Implement the `Nano.fs.*` API to expose VFS operations to JavaScript code running in isolates. This provides an explicit, NANO-specific API for filesystem operations.

## Context
- @../../../src/vfs/mod.rs — FileSystem struct with async methods
- @../../../src/vfs/isolate.rs — IsolateVfs with namespace isolation
- @../../../src/runtime/apis.rs — Pattern for JS API bindings
- @../../../src/v8/isolate.rs — NanoIsolate with VFS integration

## Current State
The VFS infrastructure exists with:
- `IsolateVfs` with namespace isolation
- File operations: read, write, exists, delete
- Path validation and security
- Error types matching Node.js semantics

What's missing:
- JavaScript bindings to expose these operations to JS code
- The `Nano` global object with `fs` namespace
- Promise-based async API for JS

## Success Criteria
1. `Nano.fs.readFile(path)` returns Promise<Uint8Array>
2. `Nano.fs.writeFile(path, data)` returns Promise<void>
3. `Nano.fs.exists(path)` returns Promise<boolean>
4. `Nano.fs.delete(path)` returns Promise<void>
5. All methods properly validate paths and throw JS errors

## Tasks

### Task 1: Create Nano.fs bindings module
**Type:** auto
**File:** src/runtime/vfs_bindings.rs (new)

Create a new module for VFS JavaScript bindings:

1. Create `src/runtime/vfs_bindings.rs`
2. Implement `bind_nano_fs()` function that:
   - Creates the `Nano` global object if it doesn't exist
   - Creates the `fs` sub-object with methods: readFile, writeFile, exists, delete
   - Each method takes path argument and returns a Promise
3. Methods should:
   - Accept path as string
   - For writeFile: accept data as string, Uint8Array, or ArrayBuffer
   - Return Promise that resolves with result or rejects with error
4. Error handling:
   - Convert VfsError to JS Error with appropriate message and code
   - Use Node.js-style error messages (e.g., "ENOENT: no such file or directory")

### Task 2: Add to RuntimeAPIs
**Type:** auto
**File:** src/runtime/apis.rs

1. Add `mod vfs_bindings;` to runtime module structure
2. In `RuntimeAPIs::bind_all()`, add call to `Self::bind_nano_fs(scope, context, isolate_vfs)`
3. Create new method `bind_nano_fs()` that delegates to vfs_bindings

### Task 3: Update mod.rs
**Type:** auto
**File:** src/runtime/mod.rs

Add `pub mod vfs_bindings;` to expose the module.

### Task 4: Modify context setup to pass VFS
**Type:** auto
**File:** src/v8/context.rs or appropriate handler setup

Ensure that when a context is created for request handling:
1. The IsolateVfs reference is available
2. Nano.fs methods have access to the VFS for the current isolate
3. The VFS reference is stored in a way accessible to JS callbacks (e.g., slot in context or isolate data)

### Task 5: Implement VFS access for JS callbacks
**Type:** auto
**File:** src/runtime/vfs_bindings.rs

1. Use V8 external storage or context slots to store IsolateVfs reference
2. Implement getter that retrieves VFS reference from callback context
3. Handle cases where VFS is not available (should not happen in production)

### Task 6: Write unit tests
**Type:** auto
**File:** src/runtime/vfs_bindings.rs (tests module)

Create tests:
1. Test Nano.fs.writeFile() and readFile() roundtrip
2. Test Nano.fs.exists() returns correct values
3. Test Nano.fs.delete() removes files
4. Test error handling (ENOENT for missing files)
5. Test path validation (traversal blocked)
