# Phase 12 Plan 02: Node.js fs Polyfill

## Objective
Implement a Node.js-compatible `fs` module polyfill that tricks Node.js applications into using the VFS backend. This enables compatibility with apps that use `require('fs')` or `import fs from 'fs'`.

## Context
- @../../../src/runtime/vfs_bindings.rs — Nano.fs API from Plan 01
- @../../../src/v8/script.rs — Script execution and module loading
- @../../../src/runtime/apis.rs — API binding patterns

## Current State
From Plan 01:
- Nano.fs.* API exists with async methods
- RuntimeAPIs can bind new APIs to global scope

What's missing:
- require() hook to intercept 'fs' module requests
- Synchronous versions of fs methods (readFileSync, writeFileSync, existsSync)
- ES module import support for 'fs'
- Node.js-compatible fs module object

## Success Criteria
1. `const fs = require('fs')` returns polyfill object
2. `fs.readFileSync('/path')` returns file contents
3. `fs.writeFileSync('/path', data)` writes file contents
4. `fs.existsSync('/path')` returns boolean
5. `import fs from 'fs'` works with ES modules
6. Polyfill routes to same VFS backend as Nano.fs

## Tasks

### Task 1: Create Node.js fs polyfill module
**Type:** auto
**File:** src/runtime/fs_polyfill.rs (new)

Create a JavaScript polyfill that:
1. Exposes Node.js-compatible fs API:
   - readFileSync(path, [options])
   - writeFileSync(path, data, [options])
   - existsSync(path)
   - readFile(path, [options], callback) — async version
   - writeFile(path, data, [options], callback) — async version
   - exists(path, callback) — async version
2. Routes to Nano.fs.* internally or directly to VFS
3. Accepts string or Buffer/Uint8Array for data
4. Returns string or Buffer based on encoding option

### Task 2: Implement require() hook
**Type:** auto
**File:** src/v8/script.rs or new src/v8/module.rs

1. Implement custom module loader that:
   - Intercepts `require('fs')` calls
   - Returns the fs polyfill object instead of looking for module
   - Handles the case where module is not found gracefully
2. Use V8's module API or FunctionCallback for require()
3. Store the polyfill object in a way that's cached per-context

### Task 3: Implement ES module import hook
**Type:** auto
**File:** src/v8/module.rs (new) or extend script.rs

1. Create V8 module loader that:
   - Handles `import fs from 'fs'` syntax
   - Returns the fs polyfill as the module namespace
2. Use V8's Module API for ES module resolution
3. Support both `import fs from 'fs'` and `import { readFileSync } from 'fs'`

### Task 4: Implement sync VFS operations
**Type:** auto
**File:** src/vfs/mod.rs (add sync methods)

Add synchronous versions of VFS methods:
1. `FileSystem::read_sync()` — blocks on async runtime
2. `FileSystem::write_sync()` — blocks on async runtime
3. `FileSystem::exists_sync()` — blocks on async runtime
4. Use `tokio::runtime::Handle::current().block_on()` for async-to-sync bridge
5. Or use a dedicated sync runtime if needed

### Task 5: Wire polyfill into context setup
**Type:** auto
**File:** src/runtime/apis.rs

1. In `RuntimeAPIs::bind_all()`, add call to bind fs polyfill
2. Make require() available globally
3. Ensure polyfill is available both via require() and as a global fallback

### Task 6: Write compatibility tests
**Type:** auto
**File:** tests/fs_polyfill_tests.rs (new)

Create tests that verify Node.js compatibility:
1. Test `require('fs').readFileSync()` works
2. Test `require('fs').writeFileSync()` works
3. Test `require('fs').existsSync()` works
4. Test async `fs.readFile()` with callback
5. Test ES module import syntax
6. Test error codes match Node.js (ENOENT, EACCES, etc.)
