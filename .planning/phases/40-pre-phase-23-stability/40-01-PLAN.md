<objective>
Fix the root cause of "cumulative None returns after CPU timeout": CpuTimeoutGuard::drop()
does not call cancel_terminate_execution(), leaving the V8 terminate flag set until the
isolate is recycled. Every subsequent handler call on the poisoned isolate returns None.

Purpose: Confirmed timing chain (test report 2026-05-17):
  default cpu_time_ms = 50ms  (AppLimits::default, src/config/app.rs:84)
  cold-start latency  ≈ 55ms  (first request on fresh isolate, test report §9.1)
  → terminate_execution() fires on every worker's first request
  → CpuTimeoutGuard::drop() does NOT call cancel_terminate_execution()
  → terminate flag persists on isolate
  → request 5 (first recycled worker) → handler_local.call() → None immediately
  → HTTP 500 "JS exception: null" for all subsequent requests on that worker

This explains exactly why endurance tests show 4/15 pass rate (27%): requests 1-4 use
workers 0-3 (each first request), requests 5-15 hit recycled workers all with sticky
terminate flags.

The current Drop impl joins the timer thread and clears the AtomicPtr but does NOT call
the V8 cancel API. This single missing call is the primary root cause of STAB-01 and
STAB-02 failures.

Output: src/data_plane.rs with cancel_terminate_execution() in CpuTimeoutGuard::drop().
</objective>

SAFETY invariant: TERMINATION_ISOLATE_PTR is non-null while CpuTimeoutGuard is alive
(set in ::new, cleared in ::drop). Reading the pointer in ::drop before clearing it
is safe under this invariant.
