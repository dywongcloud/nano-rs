## Structure:

### 1. Summary
"NANO provides approximately 5% Node.js API compatibility. It is NOT a Node.js runtime."
- It IS a WinterTC-compatible edge runtime
- It does NOT support npm packages requiring Node.js internals

### 2. What IS Supported
List implemented APIs:
- WinterTC Minimum Common API:
  - Request, Response, Headers, URL
  - TextEncoder, TextDecoder
  - console (basic)
  - crypto.getRandomValues
  - crypto.subtle (AES-GCM, HMAC)
  - ReadableStream, WritableStream
  - AbortController
  - performance.now()
  - structuredClone
  - Blob, FormData, DOMException

- VFS APIs (via Nano.fs or require('fs')):
  - readFile, writeFile
  - Limited fs compatibility

### 3. What is NOT Supported (Explicit List)
- process (no process.env, process.argv, etc.)
- require() (use ESM import instead)
- Buffer (use Uint8Array)
- module, exports, __filename, __dirname
- os, path, events, stream (Node.js versions)
- http, https, net, tls (use fetch() instead)
- fs (partial - only through VFS)
- util, url, querystring (use standard Web APIs)
- timers (setTimeout works, but implementation differs)

### 4. Framework Compatibility
| Framework | Status | Notes |
|-----------|--------|-------|
| Hono.js | ✅ Working | WinterTC-native, no Node.js deps |
| Next.js (static export) | ⚠️ Limited | May work with simple static sites |
| Next.js (SSR) | ❌ Not supported | Requires Node.js APIs |
| Astro (static) | ⚠️ Limited | Islands architecture requires verification |
| Express.js | ❌ Not supported | Requires Node.js http module |

### 5. Bundle Requirements
- Must use ES modules (export default)
- Must not import Node.js built-in modules
- Must use WinterTC APIs only
- Must bundle all dependencies (no npm resolution)

## docs/migration-guides/require-to-import.md
```markdown
# Migrating from require() to ESM

## Before (Node.js)
```javascript
const fs = require('fs');
const path = require('path');
const { helper } = require('./utils');
```

## After (NANO)
```javascript
// Use ESM imports
import { readFile } from 'node:fs'; // Won't work - see VFS guide instead
import { helper } from './utils.js'; // Note: .js extension required

// Or use standard Web APIs
const data = await Nano.fs.readFile('/data/file.txt');
```

## Key Differences
1. ESM requires file extensions: `./utils.js` not `./utils`
2. No __dirname - use relative paths or configuration
3. Third-party packages must be bundled (no npm resolution)
```

## docs/migration-guides/buffer-to-uint8array.md
```markdown
# Migrating from Buffer to Uint8Array

## Before (Node.js)
```javascript
const buf = Buffer.from('Hello');
const str = buf.toString('utf8');
const base64 = buf.toString('base64');
```

## After (NANO)
```javascript
// Text encoding
const encoder = new TextEncoder();
const uint8 = encoder.encode('Hello');

const decoder = new TextDecoder();
const str = decoder.decode(uint8);

// Base64
const base64 = btoa(String.fromCharCode(...uint8));
```

## Conversion Table
| Buffer | Uint8Array |
|--------|------------|
| Buffer.from(str) | new TextEncoder().encode(str) |
| buf.toString() | new TextDecoder().decode(buf) |
| Buffer.alloc(n) | new Uint8Array(n) |
| Buffer.concat([a,b]) | new Uint8Array([...a, ...b]) |
```

## docs/migration-guides/process-env-alternatives.md
```markdown
# Alternatives to process.env

## Problem
NANO does not provide process or process.env.

## Solutions

### 1. Build-time Configuration
Bundle config at build time:
```javascript
// config.js - processed by bundler
export const API_KEY = process.env.API_KEY; // bundler substitutes

// Or use import.meta.env with Vite/etc
export const API_KEY = import.meta.env.VITE_API_KEY;
```

### 2. Runtime Configuration via VFS
Store config in VFS:
```javascript
// Write config during deploy
await Nano.fs.writeFile('/config/env.json', JSON.stringify({
  API_KEY: 'secret-key'
}));

// Read at runtime
const config = JSON.parse(
  await Nano.fs.readFile('/config/env.json')
);
```

### 3. Request-based Configuration
Pass config in request headers or query params for per-request overrides.
```
