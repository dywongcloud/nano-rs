## Structure:

### 1. Executive Summary
"NANO's per-worker state isolation is a security feature, not a limitation."
- Each worker thread has its own V8 isolate
- Context reset clears state between requests (by design)
- This prevents data leakage between tenants and requests

### 2. The "Bug #7" Explanation

**Reported:** "State doesn't persist between requests"
**Status:** ✅ By Design - Not a Bug

**Why This Is Intentional:**
1. **Security Isolation** - Prevents cross-tenant data leakage
2. **Request Purity** - Each request starts with clean state
3. **Fault Containment** - One request's state can't corrupt another
4. **Multi-tenancy** - Shared process requires strict isolation

**The Architecture:**
```
Request 1 → Worker A (Isolate 1) → Context Reset → Request 2 (clean state)
Request 2 → Worker B (Isolate 2) → Context Reset → Request 3 (clean state)
```

### 3. How Context Reset Works

**Technical Details:**
- Each request runs in a fresh V8 context
- Global object is recreated (~5ms operation)
- Previous request's globals are garbage collected
- Heap is NOT deallocated (efficiency)

**Timing:**
- Context reset: ~5ms
- Full isolate creation: ~50-100ms
- Tradeoff: Speed vs state persistence

### 4. Worker Pool Isolation

**Affine Routing:**
- Same hostname → Same worker (cache locality)
- Different hostnames → May share worker pool
- Workers never share isolates

**Thread Safety:**
- Isolates are thread-local
- Never moved between threads
- Thread-local storage enforces this

### 5. Implications for App Developers

**What This Means:**
- ❌ Global variables reset between requests
- ❌ In-memory caches don't persist
- ❌ Module-level state is cleared
- ✅ Pure functions work perfectly
- ✅ No memory leaks between requests

**Real-World Example:**
```javascript
// DON'T: This counter resets every request
let counter = 0;
export default {
  fetch(req) {
    counter++;
    return new Response(`Count: ${counter}`);
  }
};
// Result: Always "Count: 1" because counter resets
```

## Structure:

### 1. State Options Overview

| Method | Persistence | Latency | Use Case |
|--------|-------------|---------|----------|
| VFS | Ephemeral (per-sliver) | <1ms | Config, templates |
| External DB | Persistent | 5-50ms | User data, transactions |
| Redis | Persistent | 1-5ms | Sessions, caching |
| Future KV (v2.0) | Edge-distributed | <5ms | Durable Objects alternative |

### 2. VFS State (Ephemeral)

**Best for:** Configuration, static assets, temporary files

```javascript
// Write during initialization
await Nano.fs.writeFile('/config/app.json', JSON.stringify({
  version: '1.0',
  theme: 'dark'
}));

// Read in request handler
export default {
  async fetch(req) {
    const config = JSON.parse(
      await Nano.fs.readFile('/config/app.json')
    );
    return new Response(`Theme: ${config.theme}`);
  }
};
```

**Limitations:**
- Reset on sliver restart
- Not shared across workers
- Not shared across instances

### 3. External Database (PostgreSQL, MySQL)

**Best for:** Persistent data, complex queries, transactions

```javascript
// Use fetch() to call external API
export default {
  async fetch(req) {
    // Query external API
    const response = await fetch('https://api.example.com/data');
    const data = await response.json();
    return Response.json(data);
  }
};
```

**Pattern:** Database → API → NANO app

### 4. Redis (External Cache)

**Best for:** Sessions, rate limiting, caching

```javascript
// Redis via HTTP API
export default {
  async fetch(req) {
    const sessionId = req.headers.get('Cookie')?.match(/session=([^;]+)/)?.[1];
    
    // Get session from Redis
    const session = await fetch(`https://redis.example.com/get/${sessionId}`, {
      headers: { 'Authorization': 'Bearer ' + REDIS_TOKEN }
    }).then(r => r.json());
    
    return Response.json({ user: session.user });
  }
};
```

### 5. KV Store (Future v2.0)

**Planned:** Edge-distributed key-value storage

```javascript
// Future syntax (not yet available)
export default {
  async fetch(req) {
    // Low-latency edge KV
    await Nano.kv.put('counter', 42);
    const value = await Nano.kv.get('counter');
    return new Response(`Counter: ${value}`);
  }
};
```

**Timeline:** See ROADMAP.md Phase 25+

### 6. Decision Tree

```
Need state to persist?
├── No → Use pure functions
├── Yes, same request only → Local variables
├── Yes, across requests
│   ├── < 1ms latency requirement → VFS (limited scope)
│   ├── Shared across workers → External DB/Redis
│   └── Edge-distributed → Wait for KV (v2.0)
└── Sessions/caching → Redis
```

### 7. Anti-Patterns to Avoid

```javascript
// ❌ DON'T: Expect global persistence
let cache = {};
export default {
  fetch(req) {
    cache[req.url] = Date.now(); // Lost after request!
    return new Response('OK');
  }
};

// ❌ DON'T: Module-level state for request data
const userSessions = new Map();
export default {
  fetch(req) {
    userSessions.set('user', getUser(req)); // Lost!
    return new Response('OK');
  }
};
```

Focus on practical, working examples.

Create troubleshooting guide at docs/troubleshooting.md covering common issues:

## Structure:

### 1. "My Global Variables Reset Between Requests"

**Symptom:** Counter always returns 1, cache doesn't persist
**Cause:** Context reset clears globals between requests (by design)
**Solution:** Use external state (Redis, DB) or VFS

**Example Fix:**
```javascript
// BEFORE (broken)
let counter = 0;
export default {
  fetch(req) { return new Response(++counter); }
};

// AFTER (external state)
export default {
  async fetch(req) {
    const count = await fetch('https://api.example.com/counter/increment')
      .then(r => r.json());
    return Response.json({ count });
  }
};
```

### 2. "My App Works Locally But Not in NANO"

**Check:** 
- Using `require()`? Use ESM `import` instead
- Using `process.env`? Use build-time config or VFS
- Using `Buffer`? Use Uint8Array
- Using Node.js modules? Check compatibility matrix

**Debug Steps:**
1. Check console output for errors
2. Verify ESM syntax: `export default { fetch }`
3. Test with minimal handler first

### 3. "Sliver VFS Files Not Found"

**Symptoms:** 
- "File not found" when running from sliver
- App works with `--config` but not `--sliver`

**Causes:**
1. VFS not captured during sliver creation
2. Wrong path (missing `/` prefix)
3. Sliver created from wrong directory

**Fixes:**
```bash
# Verify sliver contents
nano-rs sliver inspect app.sliver

# Check VFS entries exist
# Should show: vfs/, vfs/index.js, etc.
```

### 4. "Port Already in Use"

**Error:** `AddrInUse` when starting server

**Solutions:**
```bash
# Find and kill process
lsof -i :8080
kill -9 <PID>

# Or use different port
nano-rs run --config config.json --port 8081
```

### 5. "High Memory Usage / OOM"

**Symptoms:**
- Worker killed with OOM
- Memory grows over time

**Causes:**
1. Memory limit too low for app
2. Memory leak in JavaScript
3. Large VFS files

**Fixes:**
```json
// Increase memory limit
{
  "apps": [{
    "hostname": "api.example.com",
    "memory_limit_mb": 256
  }]
}
```

### 6. "Slow Request Latency"

**Expected:** ~0.3ms with sliver
**Seeing:** 10ms+ 

**Check:**
1. Using context reset instead of sliver? (~5ms vs ~0.3ms)
2. External API calls in handler
3. Too many workers (context switching overhead)

**Optimize:**
```bash
# Use sliver for production
nano-rs run --sliver app.sliver --workers 4
```

### 7. "V8 Platform Not Initialized" / SIGSEGV

**Error:** Segfault on sliver creation or run
**Cause:** V8 platform not initialized before sliver operations
**Fix:** Fixed in v1.1+ - ensure latest version

### 8. "Cannot Import Modules"

**Error:** `Module not found` or `Import resolution failed`
**Causes:**
1. Relative import missing `.js` extension
2. Trying to import Node.js built-in
3. Package not bundled

**Fix:**
```javascript
// ❌ WRONG
import { helper } from './utils'; // Missing extension

// ✅ CORRECT  
import { helper } from './utils.js'; // Extension required
```

### 9. "Admin API Not Responding"

**Check:**
```bash
# Verify admin server is running
curl -H "X-Admin-Key: secret" http://localhost:8889/admin/health

# Check config
nano-rs run --admin-port 8889
```

### 10. Build / Compilation Issues

**Error:** V8 compilation takes hours
**Fix:** Use pre-built V8 (default)
```bash
# Don't set V8_FROM_SOURCE
cargo build --release
```

**Error:** Rust version too old
**Fix:** Update to Rust 1.70+
```bash
rustup update
```
