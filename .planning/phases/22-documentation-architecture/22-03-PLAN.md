# ADR-001: ExternalPointerTable (EPT) SIGSEGV Fix

**Status:** Accepted  
**Date:** 2026-04-19  
**Deciders:** Core Team  
**Technical Story:** EPT segment unmapping bug causes SIGSEGV on isolate teardown

## Context and Problem Statement

When creating and destroying V8 isolates rapidly (as in a multi-tenant edge runtime), we encountered random SIGSEGV crashes during isolate teardown. Investigation revealed this was caused by V8's ExternalPointerTable (EPT) segment unmapping bug, where the table's memory is freed while still being accessed.

## Decision Drivers

* Stability — Cannot have random crashes in production
* V8 integration — Must work with rusty_v8 crate
* Performance — Fix must not add significant overhead
* Safety — Prefer Rust safety guarantees

## Considered Options

1. **Strong Global Sentinel** — Keep a `v8::Global<Value>` reference per isolate that prevents EPT cleanup until explicit drop
2. **Isolate Pool** — Reuse isolates instead of creating/destroying (doesn't fix root cause)
3. **Delay Teardown** — Add sleep/delay before isolate destruction (hacky, unreliable)
4. **Patch rusty_v8** — Modify V8 bindings (maintenance burden, upstream divergence)

## Decision Outcome

Chosen option: "Strong Global Sentinel", because it prevents the crash at the root cause (EPT lifecycle) without performance impact or upstream modifications.

We create a `v8::Global<Value>` at isolate creation and store it with the isolate. This global reference keeps the isolate's EPT segments alive until we explicitly drop it during controlled teardown, preventing the use-after-free that causes SIGSEGV.

### Positive Consequences

* Eliminates random SIGSEGV crashes
* No performance overhead
* Works with unmodified rusty_v8
* Clean, deterministic cleanup

### Negative Consequences

* Slightly more complex isolate lifecycle (need to track sentinel)
* Must be documented for future maintainers (why is this random Global here?)

## Implementation Details

Location: `src/v8/platform.rs`

```rust
// Per-isolate sentinel to prevent EPT SIGSEGV
struct IsolateState {
    isolate: v8::OwnedIsolate,
    // This Global prevents EPT segment unmapping bug
    // See ADR-001 for full context
    sentinel: v8::Global<v8::Value>,
}
```

The sentinel is created after isolate initialization and dropped explicitly during controlled teardown before the isolate itself is disposed.

## Links

* Related: ADR-003 (Thread-Local Isolates) — Isolate lifecycle management
* Code: `src/v8/platform.rs`
```

# ADR-002: Context Reset for Request Isolation

**Status:** Accepted  
**Date:** 2026-04-19  
**Deciders:** Core Team  
**Technical Story:** Need per-request isolation without 50-100ms isolate creation overhead

## Context and Problem Statement

In a multi-tenant edge runtime, each HTTP request must be isolated from previous requests to prevent data leakage between tenants. We need a mechanism to:
1. Reset JavaScript global state between requests
2. Maintain performance for high-throughput scenarios
3. Ensure security isolation

The naive approach (create new isolate per request) takes 50-100ms, which is unacceptable for edge latency requirements.

## Decision Drivers

* Latency — Target <10ms request handling overhead
* Isolation — No data leakage between requests
* Throughput — Support 1000+ RPS per core
* Security — Strong isolation guarantees

## Considered Options

1. **Context Reset** — Reset V8 context between requests (~5ms), keep isolate
2. **Fresh Isolate** — Create new isolate per request (~50-100ms)
3. **Isolate Pool** — Pre-warm isolates, checkout/checkin pattern
4. **Realm per Request** — Use V8 Realms (experimental in rusty_v8)

## Decision Outcome

Chosen option: "Context Reset", because it provides adequate isolation with 10x better latency than fresh isolates.

Context reset (~5ms) clears the JavaScript global object and reinitializes built-in APIs while keeping the V8 isolate (the expensive part). This provides:
- Clean global state per request
- All built-in APIs re-bound
- 10x faster than isolate creation (5ms vs 50-100ms)

### Positive Consequences

* 10x lower latency than fresh isolates
* Simpler implementation than pooling
* Deterministic behavior (no pool exhaustion issues)
* Adequate isolation for edge use case

### Negative Consequences

* Not perfect isolation (same isolate, potential V8 bugs)
* Context reset still takes ~5ms
* Memory usage grows with isolate lifetime (mitigated by limits)

## Performance Comparison

| Approach | Time | Use Case |
|----------|------|----------|
| Context Reset | ~5ms | Between requests (our approach) |
| Fresh Isolate | ~50-100ms | New app, sliver unavailable |
| Sliver Restore | ~267µs | New worker, snapshot available |

## Links

* Related: docs/COLD_START.md — Detailed performance characteristics
* Code: `src/worker/context.rs`

# ADR-003: Thread-Local Isolate Ownership

**Status:** Accepted  
**Date:** 2026-04-19  
**Deciders:** Core Team  
**Technical Story:** V8 isolates are NOT thread-safe, need affine execution model

## Context and Problem Statement

V8 isolates (the execution context for JavaScript) are explicitly NOT thread-safe. An isolate can only be used from the thread that created it. In a multi-threaded Rust server handling concurrent requests, we need a pattern to:
1. Handle concurrent requests across multiple threads
2. Respect V8's thread-safety constraints
3. Avoid cross-thread isolate migration (which causes crashes)

## Decision Drivers

* Safety — Cannot violate V8 thread requirements
* Performance — Minimize thread synchronization overhead
* Simplicity — Avoid complex ownership tracking
* Throughput — Scale with CPU cores

## Considered Options

1. **Thread-Local Ownership** — Each worker thread owns its isolates permanently
2. **Mutex Pool** — Protect isolate access with locks (deadlock risk, contention)
3. **Message Passing** — Send isolate to thread, use, send back (complex lifecycle)
4. **Single-Threaded** — One thread for all JS (bottleneck on multi-core)

## Decision Outcome

Chosen option: "Thread-Local Ownership", because it respects V8 constraints with zero runtime overhead.

Each worker thread in the WorkerPool owns its isolates for the thread's entire lifetime. Request dispatch routes to specific workers (not free-for-all), ensuring affine execution:

```
Thread 1: [Isolate A] [Isolate B] — owns both permanently
Thread 2: [Isolate C] [Isolate D] — owns both permanently
Request with hash % 4 → routed to specific thread/isolate
```

### Positive Consequences

* Zero synchronization overhead (no locks on hot path)
* Respects V8 thread-safety constraints
* Simple mental model: thread owns isolate
* Cache-friendly (worker always uses same isolates)

### Negative Consequences

* Cannot dynamically balance load across threads
* Worker with slow request blocks that thread's isolates
* Requires careful worker pool sizing

## Implementation

Location: `src/worker/pool.rs`, `src/worker/queue.rs`

WorkerPool spawns N threads, each with thread_local isolates. WorkQueue uses bounded MPSC channel per worker for dispatch.

## Links

* Related: ADR-001 (EPT Fix) — Isolate lifecycle safety
* Related: ADR-002 (Context Reset) — What happens in the isolate
* Code: `src/worker/pool.rs`
```

# ADR-004: Virtual File System Abstraction

**Status:** Accepted  
**Date:** 2026-04-20  
**Deciders:** Core Team  
**Technical Story:** Need filesystem API for JS without direct OS filesystem access

## Context and Problem Statement

JavaScript applications need file I/O for:
- Reading configuration
- Writing logs/state
- Static file serving

But direct filesystem access in a multi-tenant runtime is dangerous:
- Path traversal attacks (../../../etc/passwd)
- Cross-tenant data access
- Host filesystem pollution

We need a filesystem abstraction that provides:
1. Security isolation between tenants
2. Snapshot support (for slivers)
3. Pluggable storage backends

## Decision Drivers

* Security — Prevent filesystem escape
* Isolation — Per-tenant filesystem namespaces
* Portability — Works with sliver snapshots
* Flexibility — Multiple storage backends

## Considered Options

1. **Direct OS Filesystem** — Simple, but insecure (rejected)
2. **VFS Abstraction** — Per-isolate namespaces, pluggable backends (chosen)
3. **chroot per Isolate** — Linux-specific, complex, overhead
4. **FUSE Filesystem** — External process, too heavy

## Decision Outcome

Chosen option: "VFS Abstraction", because it provides security, isolation, and flexibility.

VFS (`src/vfs/`) provides:
- Per-isolate namespaces: `{hostname}::/path/to/file`
- Path validation: Rejects `..`, null bytes, absolute paths
- Pluggable backends: Memory, Disk, S3
- Snapshot support: Can serialize entire VFS state

### Backend Strategy

| Backend | Use Case | Status |
|---------|----------|--------|
| Memory | Fast, ephemeral, sliver snapshots | Default |
| Disk | Persistence across restarts | Optional |
| S3 | Cloud-native, scalable | Feature-gated |

### Positive Consequences

* Strong security isolation
* Sliver snapshots capture full filesystem state
* Multiple storage options for different deployments
* Consistent API regardless of backend

### Negative Consequences

* More complex than direct filesystem access
* Memory backend uses RAM (mitigated by quotas)
* Some Node.js fs APIs not supported (async only)

## Links

* Related: docs/VFS.md — Detailed VFS documentation
* Code: `src/vfs/`

# ADR-005: Rust Crypto Implementation Strategy

**Status:** Accepted  
**Date:** 2026-04-19  
**Deciders:** Core Team  
**Technical Story:** Implement WebCrypto via Rust crates vs using V8's built-in crypto

## Context and Problem Statement

WebCrypto API (`crypto.subtle`) needs implementation for:
- SHA-256/512 hashing
- AES-GCM encryption/decryption
- HMAC signing/verification
- Key generation and import/export

V8 has built-in crypto.subtle C++ implementation, but:
1. Complex C++ bindings required
2. Less control over algorithm implementations
3. Harder to audit for security

Alternative: Use Rust crypto crates (`ring`, `aes_gcm`) and expose via V8 bindings.

## Decision Drivers

* Security — Prefer auditable Rust code
* Control — Full control over algorithms
* Safety — Rust memory safety vs C++
* Maintenance — Easier to update dependencies

## Considered Options

1. **Rust Crypto (ring, aes_gcm)** — Pure Rust, safe, auditable (chosen)
2. **V8 Built-in** — Less code, but complex bindings
3. **OpenSSL via FFI** — Industry standard, but unsafe FFI
4. **Rustls** — TLS-focused, not general crypto

## Decision Outcome

Chosen option: "Rust Crypto", because it provides memory safety, auditability, and full control.

Implementation uses:
- `ring` crate: SHA-256, SHA-512, HMAC
- `aes_gcm` crate: AES-GCM encryption
- `zeroize` crate: Secure key material erasure

All operations happen in Rust, results passed to V8 as ArrayBuffers.

### Positive Consequences

* Memory-safe crypto operations
* Easy to audit (pure Rust)
* Full algorithm control
* Simple dependency updates
* Consistent with Rust-first architecture

### Negative Consequences

* More initial implementation work than using V8 built-in
* Must re-implement WebCrypto API surface
* Async operations require Promise bridging

## Algorithm Support

| Algorithm | Status | Crate |
|-----------|--------|-------|
| SHA-256 | ✅ | ring |
| SHA-512 | ✅ | ring |
| AES-GCM | ✅ | aes_gcm |
| HMAC | ✅ | ring |
| RSA | 🚧 Planned | rsa (v2.0) |
| ECDSA | 🚧 Planned | p256 (v2.0) |

# ADR-006: Sliver Snapshot Format

**Status:** Accepted  
**Date:** 2026-04-20  
**Deciders:** Core Team  
**Technical Story:** Design portable format for isolate snapshots

## Context and Problem Statement

For edge deployment, we need:
1. Fast isolate startup (~1ms target)
2. Portable snapshots (deploy anywhere)
3. Full state capture (heap + filesystem)
4. Version-agnostic format (survive V8 upgrades)

Challenge: V8 heap snapshots are version-specific. A snapshot created with V8 12.x won't work with 12.y if the heap layout changed.

## Decision Drivers

* Portability — Same sliver works on any NANO version
* Speed — Sub-millisecond restore time
* Completeness — Capture heap + VFS + metadata
* Inspectability — Can inspect without NANO (debugging)

## Considered Options

1. **Tar + Opaque Heap Blob** — Standard tar with V8 heap as opaque bytes (chosen)
2. **Pure V8 Snapshot** — Fast, but version-specific
3. **Custom Binary Format** — Efficient, but not inspectable
4. **JSON + base64** — Human-readable, but huge and slow

## Decision Outcome

Chosen option: "Tar + Opaque Heap Blob", because it balances portability, inspectability, and performance.

Sliver format structure:
```
sliver.tar
├── meta.json       # Metadata (host, created, version)
├── heap.bin        # Opaque V8 heap blob (version-specific)
├── manifest.txt    # File listing
└── vfs/            # Virtual filesystem
    ├── file1.js
    ├── file2.json
    └── ...
```

**Why tar?**
- Human-inspectible (`tar -tf`)
- Standard tool support
- Streaming friendly
- Unix philosophy

**Why opaque heap?**
- V8 handles version-specific details
- We just pass it through
- Survives V8 upgrades (recreated, not migrated)

### Positive Consequences

* Portability: tar works everywhere
* Inspectability: can debug with standard tools
* Version safety: opaque blob forces recreation on V8 changes
* ~267µs restore time (measured)

### Negative Consequences

* Larger than pure binary format (compression planned)
* Heap blob not human-readable
* Requires tar library (additional dependency)

## Performance

| Metric | Value |
|--------|-------|
| Sliver creation | ~50-100ms |
| Sliver restore | ~267µs |
| Size | ~10-100KB (typical) |
| Portable | Yes (opaque heap)

## Links

* Related: ADR-004 (VFS) — VFS state captured in sliver
* Related: docs/SLIVER.md — User-facing sliver documentation
* Related: src/sliver/FORMAT.md — Format specification
* Code: `src/sliver/`
```

# ADR-007: ESM Module Execution Strategy

**Status:** Accepted (v1.x), Planned Revision (v2.0)  
**Date:** 2026-04-21  
**Deciders:** Core Team  
**Technical Story:** Support ES modules (`export default { fetch }`) with V8

## Context and Problem Statement

Modern JavaScript uses ES modules:
```javascript
export default {
  async fetch(request) {
    return new Response("Hello");
  }
};
```

But classic scripts use:
```javascript
export default {
  async fetch(request) {
    return new Response("Hello");
  }
};
// Note: In classic mode, this was wrapped by test harness
```

V8 has two ways to execute ESM:
1. **Full Module API** — Proper V8 Module loading with import resolution
2. **Script Compilation** — Classic Script::compile (what we started with)

We need ESM support for frameworks like Hono.js, Next.js, Astro.

## Decision Drivers

* Compatibility — Must run modern framework bundles
* Complexity — Balance implementation effort
* Performance — Fast compilation/execution
* Standards — Eventually full ESM support

## Considered Options

1. **Full V8 Module API** — Proper ESM with imports/exports (complex, deferred)
2. **Transformation** — Transform ESM to classic script syntax (chosen for v1.x)
3. **Hybrid** — Detect and use appropriate method
4. **Wait for rusty_v8** — Defer until Module API stabilizes

## Decision Outcome

Chosen option for v1.x: "Transformation"

Transform ESM syntax to classic script-compatible code:
```javascript
// Original ESM
export default { fetch: handler };

// Transformed to classic script
var __nano_exports = { fetch: handler };
__nano_module.exports = __nano_exports;
```

This provides:
- Immediate framework compatibility
- Works with existing Script::compile infrastructure
- No V8 Module API complexity

### Positive Consequences

* Works immediately with Hono.js, Next.js, Astro
* Minimal implementation complexity
* Fast compilation
* Backward compatible with classic scripts

### Negative Consequences

* Not "real" ESM (no import resolution)
* Requires bundling (user must bundle dependencies)
* Future: Full Module API still needed for dynamic imports

## Future: Full Module API (v2.0)

Planned for v2.0 Phase 28:
- Proper V8::Module API with Module::Instantiate
- VFS-backed import resolution
- Dynamic import() support
