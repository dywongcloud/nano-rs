## WorkerPool Architecture Analysis

### Current State (from code review)

**src/worker/pool.rs (2286 lines):**
- `WorkerPool` - Complex pool with CPU timeout, memory monitoring, eviction, OOM handling, per-tenant metrics
- `SliverWorkerPool` - Snapshot-restored isolates with same features
- Both have: `with_backend()` for custom VFS, context reset, full lifecycle management

**src/worker/queue.rs (870 lines):**
- `WorkerPool` - Simple pool for entrypoint dispatch (no sliver, no CPU timeout, no memory monitoring)
- `WorkQueue` - Manages multiple WorkerPools per hostname with affine dispatch
- Has: `with_vfs_config()` for async disk backend creation

### Key Differences

| Feature | pool.rs WorkerPool | queue.rs WorkerPool |
|---------|-------------------|---------------------|
| CPU timeout | Yes (timer-based) | No |
| Memory monitoring | Yes (post-execution) | No |
| Eviction management | Yes | No |
| OOM handling | Yes | No |
| Metrics integration | Yes | No |
| Snapshot support | Yes (SliverWorkerPool) | No |
| Async creation | No | Yes (WorkQueue) |
| Multi-pool mgmt | No | Yes (WorkQueue) |

### Decision: Option B - Separate Clearly

**Rationale:**
1. Different use cases: sliver execution vs entrypoint dispatch
2. SliverWorkerPool needs snapshot restoration (complex)
3. WorkQueue manages multiple pools per hostname (different abstraction)
4. Merging would create unmaintainable complexity

### Architecture After Refactor

```
                    WorkerPool (trait)
                         |
         +---------------+---------------+
         |                               |
SliverWorkerPool               EntrypointWorkerPool
(pool.rs)                        (queue.rs)
         |                               |
    +----+----+                   +------+------+
    |         |                   |             |
CPU timeout  Memory          WorkQueue    Async VFS
Eviction     Metrics         (multi-host) creation
```

//! ## Architecture Overview

//! This module provides multiple worker pool implementations for different use cases:

//!   - Use when: Running from sliver files (production deployment)
//!
//! - **EntrypointWorkerPool** (`queue::EntrypointWorkerPool`): For entrypoint dispatch  
//!   - Creates fresh isolates from source files
//!   - Supports async creation with custom VFS backends
//!   - Use when: Dynamic app loading, development, testing
//!
//! - **WorkQueue** (`queue::WorkQueue`): Multi-hostname pool manager
//!   - Manages multiple EntrypointWorkerPools per hostname
//!   - Affine dispatch for cache locality
//!   - Use when: Multi-tenant hosting with dynamic pool creation
//!
//! ### Trait-Based Design
//!
//! All pool types implement the [`WorkerPool`] trait for common operations:
//! - `dispatch()`: Send tasks to workers
//! - `shutdown()`: Graceful shutdown
//! - `worker_count()`, `hostname()`: Introspection
//!
//! ### Selection Guide
//!
//! | Use Case | Pool Type | Why |
//! |----------|-----------|-----|
//! | Production sliver execution | SliverWorkerPool | Fast snapshot restore |
//! | Dynamic app loading | EntrypointWorkerPool | Async creation, VFS flexibility |
//! | Multi-tenant hosting | WorkQueue | Per-hostname pool management |
//! | Testing/development | EntrypointWorkerPool | Simple, no snapshot needed |
//!
//! ### VFS Backend Configuration
//!
//! Both pool types support custom VFS backends through their constructors:
//! - `SliverWorkerPool::with_backend()` - for snapshot execution with custom storage
//! - `EntrypointWorkerPool::with_backend()` - for entrypoint dispatch with custom storage
//! - `WorkQueue::with_vfs_config()` - for async disk backend creation
//!
//! See individual module documentation for details.
