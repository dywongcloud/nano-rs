# Phase 15 Plan 04: Worker Pool Sliver Integration

## Objective
Integrate sliver restoration into the worker pool and app registry. Enable running apps from sliver files with proper worker lifecycle management.

## Context

### Current State
- WorkerPool manages thread-local isolates
- AppRegistry tracks app configurations
- HTTP router dispatches requests to apps by hostname
- Apps are currently loaded from entrypoint scripts

### Integration Points
- src/worker/pool.rs — WorkerPool manages isolates per app
- src/app/registry.rs — AppRegistry needs sliver-based app loading
- src/http/router.rs — Router resolves hostnames to apps
- src/sliver/restore.rs — VFS restoration from slivers

## Tasks

### Task 1: Add sliver loading to AppRegistry

**Type:** auto

**Read First:**
- src/app/registry.rs — Current AppRegistry implementation
- src/sliver/unpacker.rs — SliverUnpacker usage
- src/config/app.rs — AppConfig with sliver field

**Action:**
Add sliver-based app registration to AppRegistry:

1. Add `sliver_data` field to AppRegistry struct
2. Implement `register_from_sliver()` method
3. Add `get_sliver_data()` getter

The method should:
- Read the sliver file
- Unpack it using SliverUnpacker
- Extract hostname from metadata
- Create AppConfig with sliver field
- Store unpacked sliver data for workers

**Acceptance Criteria:**
- `register_from_sliver()` method added
- AppRegistry tracks sliver_data separately
- Hostname extracted from sliver metadata
- Proper error handling for invalid slivers
- Unit tests for registration

### Task 2: Create SliverWorkerPool

**Type:** auto

**Read First:**
- src/worker/pool.rs — Current WorkerPool implementation
- src/v8/snapshot.rs — restore_from_snapshot function
- src/sliver/restore.rs — VFS restoration functions

**Action:**
Create a specialized SliverWorkerPool that:
- Takes hostname, worker_count, and unpacked sliver
- Spawns workers that restore isolates from snapshots
- Restores VFS entries to each worker
- Falls back to fresh isolate if snapshot fails
- Supports graceful shutdown

**Acceptance Criteria:**
- SliverWorkerPool struct created
- `new()` constructor takes hostname, worker_count, unpacked_sliver
- `start()` method spawns workers with restored isolates
- Workers restore VFS entries before accepting requests
- Fallback to fresh isolate if snapshot fails
- `shutdown()` method for graceful shutdown
