# Phase 15 Plan 01: CLI --sliver Flag and Config Integration

## Objective
Add the `--sliver` flag to the `nano-rs run` command and extend the configuration system to support sliver-based app loading.

## Context

### Current State
The `run` command currently only accepts a `--config` option for traditional app loading. Sliver support requires:
1. A `--sliver` flag for direct sliver file execution
2. A `sliver` field in AppConfig for config-based loading
3. Validation for sliver-based app configurations

### Integration Points
- src/main.rs — Add `--sliver` flag parsing
- src/config/app.rs — Add sliver field and validation
- src/cli/mod.rs — May need updates for command routing

## Tasks

### Task 1: Add sliver field to AppConfig

**Type:** auto

**Read First:**
- src/config/app.rs — Current AppConfig structure
- src/sliver/metadata.rs — SliverMetadata structure for reference

**Action:**
Add a `sliver` field to `AppConfig` that allows specifying a sliver file path instead of an entrypoint:

```rust
/// Application configuration for a single hosted app
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
#[serde(deny_unknown_fields)]
pub struct AppConfig {
    /// Hostname this app responds to (e.g., "api.example.com")
    pub hostname: String,

    /// Path to the entry point JavaScript file (required unless sliver is set)
    #[serde(default)]
    pub entrypoint: String,

    /// Path to sliver file for snapshot-based loading (alternative to entrypoint)
    #[serde(default)]
    pub sliver: Option<String>,

    // ... rest of fields unchanged
}
```

Add validation logic:
- At least one of `entrypoint` or `sliver` must be specified
- If both are specified, `sliver` takes precedence
- If `sliver` is specified, validate the file path (security: no "..")

**Acceptance Criteria:**
- AppConfig deserializes with optional `sliver` field
- Validation rejects configs with neither entrypoint nor sliver
- Validation rejects sliver paths containing ".."
- Unit tests for new validation rules pass

### Task 2: Add --sliver flag to CLI

**Type:** auto

**Read First:**
- src/main.rs — Current CLI structure with Commands enum
- src/cli/sliver.rs — Existing sliver command patterns

**Action:**
Extend the `Run` command variant to support sliver execution:

```rust
#[derive(Debug, Subcommand)]
enum Commands {
    /// Run the NANO HTTP server (default behavior)
    Run {
        /// Configuration file path
        #[arg(short, long, value_name = "FILE")]
        config: Option<PathBuf>,
        
        /// Sliver file to run directly
        #[arg(long, value_name = "FILE", conflicts_with = "config")]
        sliver: Option<PathBuf>,
        
        /// Number of workers (when using --sliver)
        #[arg(short, long, default_value = "4")]
        workers: usize,
    },
    // ...
}
```

Update the main match logic to handle sliver execution path.

**Acceptance Criteria:**
- CLI parses `nano-rs run --sliver ./app.sliver --workers 8`
- --sliver conflicts with --config (can't use both)
- --workers only valid with --sliver
- Help text shows new options

### Task 3: Add validation for sliver-based configs

**Type:** auto

**Read First:**
- src/config/app.rs — validate_config function

**Action:**
Update `validate_config` to handle sliver-based apps:

1. Modify entrypoint validation to be optional if sliver is set
2. Add sliver path validation:
   - Non-empty path check
   - Path traversal prevention (no "..")
   - File existence check (when base_path provided)
3. Ensure at least one of entrypoint or sliver is present

Add new error messages:
- "either entrypoint or sliver must be specified"
- "sliver path cannot be empty"
- "sliver path contains '..' which is not allowed for security"

**Acceptance Criteria:**
- validate_config accepts configs with sliver but no entrypoint
- validate_config rejects configs with neither sliver nor entrypoint
- Path traversal attempts in sliver paths are rejected
- Unit tests for all new validation paths pass
