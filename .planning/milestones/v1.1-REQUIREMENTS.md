## v1.1 Requirements — Complete ✅

### Snapshot Core

- [x] **SNAP-01:** CLI `nano-rs snapshot create <hostname>` produces tar archive  
  **Outcome:** ✅ Implemented as `nano-rs sliver create <hostname> --output <file.sliver>`

- [x] **SNAP-02:** Snapshot tar contains V8 isolate heap + VFS state + metadata  
  **Outcome:** ✅ Archive structure: `meta.json`, `heap.bin`, `vfs/` directory, `manifest.txt`

- [x] **SNAP-03:** CLI `nano-rs run --snapshot app-v1.tar` restores isolate from snapshot  
  **Outcome:** ✅ Implemented as `nano-rs run --sliver <file.sliver>`

- [x] **SNAP-04:** Restored isolate resumes execution with preserved state  
  **Outcome:** ✅ SliverWorkerPool restores VFS and heap before accepting tasks

- [x] **SNAP-05:** Snapshot format is tar-based with simple structure  
  **Outcome:** ✅ Standard tar format, inspectable with `tar -tf`

- [x] **SNAP-06:** Snapshots are opaque blobs (version-agnostic, no embedded versioning)  
  **Outcome:** ✅ `heap.bin` is opaque blob, format uses string versions

- [x] **SNAP-07:** Multiple snapshots can coexist (versioned by filename)  
  **Outcome:** ✅ `sliver list` shows all `.sliver` files with metadata

### VFS Core

- [x] **VFS-01:** VFS module at `src/vfs/` with in-memory storage  
  **Outcome:** ✅ MemoryBackend with DashMap for concurrent access

- [x] **VFS-02:** Per-isolate filesystem namespace (no cross-app access)  
  **Outcome:** ✅ IsolateVfs with `{hostname}::{path}` prefix isolation

- [x] **VFS-03:** JS API `Nano.fs.readFile(path)` reads file contents  
  **Outcome:** ✅ Returns `Promise<Uint8Array | string>`

- [x] **VFS-04:** JS API `Nano.fs.writeFile(path, data)` writes file contents  
  **Outcome:** ✅ Accepts `string | Uint8Array | ArrayBuffer`

- [x] **VFS-05:** JS API `Nano.fs.exists(path)` checks file existence  
  **Outcome:** ✅ Returns `Promise<boolean>`

- [x] **VFS-06:** Optional disk backing for VFS persistence  
  **Outcome:** ✅ DiskBackend with atomic writes

- [x] **VFS-07:** Optional S3-compatible object storage backend  
  **Outcome:** ✅ S3Backend feature-gated via `vfs-s3`

- [x] **VFS-08:** VFS state included in snapshot serialization  
  **Outcome:** ✅ VFS entries stored under `vfs/` prefix in sliver

### Node.js Compatibility Layer

- [x] **NODE-01:** `require('fs')` resolves to VFS polyfill  
  **Outcome:** ✅ Global `require()` function returns fs polyfill

- [x] **NODE-02:** `fs.readFileSync()` routes to `Nano.fs.readFile()`  
  **Outcome:** ✅ Sync method with tokio block_on

- [x] **NODE-03:** `fs.writeFileSync()` routes to `Nano.fs.writeFile()`  
  **Outcome:** ✅ Sync method implemented

- [x] **NODE-04:** `fs.existsSync()` routes to `Nano.fs.exists()`  
  **Outcome:** ✅ Sync method returning boolean

- [x] **NODE-05:** ES module `import fs from 'fs'` resolves to VFS polyfill  
  **Outcome:** ✅ ES module imports work

- [x] **NODE-06:** Common error codes (ENOENT, EACCES) match Node.js semantics  
  **Outcome:** ✅ Error.code property matches Node.js (ENOENT, EINVAL, etc.)

### Performance & Migration

- [x] **PERF-01:** Cold start from snapshot achieves ~1-2ms latency  
  **Outcome:** ✅ ~267µs achieved (3.7x better than target)

- [x] **MIGRATE-01:** Snapshot can be transferred between NANO instances  
  **Outcome:** ✅ Portable tar format, verified with migration tests

### CLI & Management

- [x] **CLI-01:** `nano-rs snapshot list` shows available snapshots  
  **Outcome:** ✅ `nano-rs sliver list [--verbose]`

- [x] **CLI-02:** `nano-rs snapshot delete <name>` removes snapshot  
  **Outcome:** ✅ `nano-rs sliver delete <name> [--force]`

## Out of Scope (Valid Decisions)

| Feature | Reason | Status |
|---------|--------|--------|
| Delta/differential snapshots | Complex, defer to v1.2+ | 📋 Planned |
| Live migration (running isolates) | Requires freeze/thaw, significant complexity | 📋 Deferred |
| npm package resolution | Apps remain single-file, bundling is user responsibility | ❌ Out of scope |
| TypeScript/JSX transpilation | User must bundle beforehand | ❌ Out of scope |
| Native module support | Only pure JS/WinterCG APIs | ❌ Out of scope |
| Subprocess spawning from JS | Security/scope constraint | ❌ Out of scope |
| Built-in horizontal clustering | External load balancer sufficient | ❌ Out of scope |
| VFS directory operations (mkdir, readdir) | Defer to v1.2, focus on file I/O first | 📋 Planned |
| VFS file watching | Complex, defer to v2.0 | 📋 Deferred |
