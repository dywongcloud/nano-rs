# Auto-Sliver Architecture

## Overview

Auto-sliver provides transparent optimization for JavaScript cold starts by automatically generating and caching V8 heap snapshots after the first successful request.

## Hot-Loading Architecture

Running nano-rs instances can detect and use slivers generated by:
- Other processes (multi-process coordination)
- Earlier in the same process's lifecycle
- Previous server restarts

### Lock File Mechanism

```
Cache Directory Structure:
~/.nano/sliver-cache/
└── example.com/
    ├── <hash>.sliver      # The actual sliver snapshot
    ├── <hash>.lock        # Generation lock (prevents duplicates)
    └── ...

Lock Acquisition:
1. Process 1: create_new(<hash>.lock) → Success → Generate sliver
2. Process 2: create_new(<hash>.lock) → Fail (already exists) → Wait
3. Process 2: Wait up to 5 seconds for sliver to appear
4. Process 2: If sliver appears → Use it (hot-load)
5. Process 2: If timeout → Release stale lock, retry
```

## 4-Process Pool Architecture

### Configuration

```yaml
# Load balancer (nginx/haproxy) configuration
upstream nano_workers {
    least_conn;  # Route to least busy worker
    server 127.0.0.1:8081;
    server 127.0.0.1:8082;
    server 127.0.0.1:8083;
    server 127.0.0.1:8084;
}

server {
    listen 80;
    location / {
        proxy_pass http://nano_workers;
    }
}
```

## Performance Characteristics

### Cold Start Times

| Scenario | Time | Description |
|----------|------|-------------|
| Source Compilation | 50-100ms | First request, no cache |
| Sliver Restore | 5-10ms | Subsequent requests (10x faster) |
| Sliver Generation | 100-500ms | Background, doesn't block requests |
| Hot-Load Detection | 0-100ms | Check cache, load if available |

### Cache Invalidation Triggers

1. **Source File Modified**: Hash changes → Regenerate
2. **V8 Version Update**: Major version mismatch → Regenerate all
3. **Sliver Corruption**: Failed unpack → Delete and regenerate
4. **Manual Clear**: Admin API or CLI → Clear cache

### Storage Requirements

```
Per-App Sliver Size:
- Heap Snapshot: 100KB - 50MB (depends on app complexity)
- VFS Files: App-dependent (static assets, dependencies)
- Metadata: ~1KB

Example for typical web app:
- Entrypoint: /app/index.js (5KB)
- Dependencies: node_modules/ (10MB)
- Static Assets: public/ (2MB)
- Generated Sliver: ~15MB
```

## Operational Considerations

### Deployment

1. **First Deployment**: All processes compile from source on first request
2. **Steady State**: All processes hot-load from shared cache
3. **Rolling Update**: New processes hot-load existing slivers from old processes
4. **Cache Warming**: Optional pre-generation via CLI `nano sliver create`

### Monitoring

```rust
// Get cache statistics
let stats = cache.stats()?;
println!("Slivers: {}, Total Size: {} MB", 
    stats.sliver_count,
    stats.total_size_bytes / 1024 / 1024
);
```

### Cleanup

```rust
// Remove slivers older than 30 days
cache.cleanup(Duration::from_secs(30 * 24 * 60 * 60))?;
```

## Future Enhancements

1. **Distributed Cache**: Redis/S3 backend for multi-server deployments
2. **Pre-generation**: CLI command to generate all slivers at deploy time
3. **Delta Updates**: Incremental sliver updates for faster regeneration
4. **Compression**: LZ4/zstd compression for large slivers
5. **Tiered Storage**: Hot slivers in memory, cold on disk
